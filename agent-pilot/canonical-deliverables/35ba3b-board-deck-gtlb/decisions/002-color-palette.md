# ADR-002: Color Palette — Teal for GitLab, Red/Blue/Green for Scenarios

**Date:** 2026-04-26
**Status:** Accepted

## Context
The deck needs a color palette that is distinctive but not decorative. Colors must reinforce the data, not distract from it.

## Decision
- **GitLab:** Teal (#0D9488) — signals growth, technology, trust
- **Bear scenario:** Red (#DC2626) — universally understood as downside
- **Base scenario:** Blue (#2563EB) — neutral, professional
- **Bull scenario:** Green (#16A34A) — universally understood as upside
- **Competitors:** Distinct colors, consistent across all slides
- **Background:** White (#FFFFFF) with subtle grid lines (#F3F4F6)

## Rationale
1. **Teal for GitLab** — Differentiates from the red/blue/green scenario colors. Teal signals technology and growth without being aggressive (red) or conservative (blue).
2. **Scenario colors are universal** — Red/green for bear/bull is understood by every board member regardless of background. Blue for base is neutral.
3. **No decorative colors** — Every color serves a data purpose. No purple, orange, or yellow unless they represent specific data points.
4. **Consistent across slides** — GitLab is always teal. Bear is always red. This allows board members to quickly orient themselves.

## Alternatives Considered
- **Red for GitLab:** Rejected — too aggressive, conflicts with bear scenario.
- **Blue for GitLab:** Rejected — conflicts with base scenario.
- **Custom palette:** Rejected — board members need to orient quickly. Standard colors reduce cognitive load.

## Consequences
- All charts use this palette consistently.
- The color scheme is documented in every chart script.
- Board members can quickly identify GitLab vs. competitors vs. scenarios.
